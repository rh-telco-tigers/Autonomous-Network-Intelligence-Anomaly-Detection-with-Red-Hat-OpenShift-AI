# AI Playbook Generation With watsonx

## Purpose

This document defines the Phase 8 contract for on-demand Ansible playbook generation after RCA is available.

Use this file when you need to know:

- which Kafka topic carries the generation request
- what the plain-text instruction contains
- which control-plane callback endpoint the external generator must call
- how the generated playbook shows up in the UI and becomes executable

## End-To-End Flow

1. An operator generates remediations for an incident after RCA exists.
2. The remediation view exposes an optional `Generate AI Ansible playbook with watsonx` step.
3. The UI calls `POST /incidents/{incident_id}/remediation/{remediation_id}/generate-playbook`.
4. The control-plane publishes one plain-text instruction to Kafka topic `aiops-ansible-playbook-generate-instruction`.
5. The external project consumes that instruction, runs its watsonx-backed generation flow, and produces one reviewable Ansible playbook in YAML.
6. The external project calls back into the control-plane with the generated playbook or a failure status.
7. The control-plane converts the pending generator remediation into a normal `ansible_playbook` remediation.
8. The UI shows that remediation as `AI generated`, and operators can approve and execute it like the built-in playbooks.

## Kafka Topic

- Topic name: `aiops-ansible-playbook-generate-instruction`
- Key: correlation id generated by the control-plane
- Value: plain-text instruction

The control-plane config uses:

- `AI_PLAYBOOK_GENERATION_KAFKA_TOPIC`
- `AI_PLAYBOOK_GENERATION_KAFKA_BOOTSTRAP_SERVERS`
- `AI_PLAYBOOK_GENERATION_ENABLED`

## Plain-Text Instruction Contract

The Kafka value is intentionally plain text so the external project can treat it as a natural-language work item.

The control-plane instruction includes:

- incident id
- project
- anomaly type
- severity
- model confidence
- workflow revision
- summarized feature signals
- RCA root cause
- RCA explanation and recommendation
- current ranked remediation titles when available
- optional operator note
- optional incident page URL
- callback URL
- correlation id
- the required callback JSON fields

The external project should treat:

- `callback_url` as authoritative
- `correlation_id` as required for the callback
- the rest of the text as generation context

## Callback Endpoint

The external project must `POST` the result to:

```text
/incidents/{incident_id}/playbook-generation/callback
```

Authentication:

- header: `x-api-key: <control-plane-api-key>`
- the key must map to a role that includes `automation`

The base URL comes from:

- `PLAYBOOK_GENERATION_CALLBACK_BASE_URL`, if set
- otherwise `CONTROL_PLANE_PUBLIC_URL`
- otherwise the internal `CONTROL_PLANE_URL`

If the external project runs outside the cluster, set `PLAYBOOK_GENERATION_CALLBACK_BASE_URL` or `CONTROL_PLANE_PUBLIC_URL` to the public control-plane Route URL.

## Callback Body

Success payload:

```json
{
  "correlation_id": "corr-123",
  "status": "generated",
  "title": "Apply AI-generated registration storm guardrail",
  "description": "Throttle ingress retries and preserve downstream stability.",
  "summary": "Generated from RCA and ranked remediation context.",
  "expected_outcome": "Retry traffic slows and registrations stabilize.",
  "preconditions": ["Review ingress namespace", "Confirm rollback note"],
  "playbook_yaml": "---\n- hosts: localhost\n  gather_facts: false\n  tasks: []\n",
  "playbook_ref": "ai_generated_playbook_corr-123",
  "action_ref": "ai_generated_playbook_corr-123",
  "provider_name": "watsonx",
  "provider_run_id": "wx-run-42",
  "metadata": {
    "model": "granite",
    "generator_version": "2026-04-06"
  }
}
```

Failure payload:

```json
{
  "correlation_id": "corr-123",
  "status": "failed",
  "provider_name": "watsonx",
  "provider_run_id": "wx-run-42",
  "error": "Prompt validation failed because no safe target could be derived."
}
```

Required rules:

- `correlation_id` must match the value from Kafka
- `status` must be `generated` or `failed`
- `playbook_yaml` is required for `generated`
- `error` should be populated for `failed`
- `action_ref` and `playbook_ref` are optional; if omitted, the control-plane creates stable defaults from the correlation id

## What The Control-Plane Does With The Callback

On `status=generated`:

- updates the pending generator remediation
- changes its type to `ansible_playbook`
- stores the returned YAML
- marks it as `AI generated`
- exposes it as a normal remediation card in the UI

On `status=failed`:

- keeps the generator remediation visible
- records the failure reason
- lets the operator retry generation from the UI

## UI Behavior

The remediation panel shows:

- an optional AI generation card while the request is pending or failed
- the Kafka correlation id and topic after the request is sent
- an `AI generated` badge on the returned playbook remediation
- the returned playbook as a normal remediation that can be approved and executed

## External Project Checklist

- consume plain-text messages from `aiops-ansible-playbook-generate-instruction`
- preserve the control-plane `correlation_id`
- generate one safe Ansible playbook in YAML
- call the control-plane callback endpoint with the result
- include `x-api-key`
- return `failed` with an explicit `error` if no safe playbook can be produced
