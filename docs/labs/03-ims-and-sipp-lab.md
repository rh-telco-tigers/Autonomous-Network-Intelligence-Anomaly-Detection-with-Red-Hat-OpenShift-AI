# Lab 03: IMS and SIPp Lab

## Objective

Deploy the IMS lab plane, generate repeatable SIP traffic, and prove that the platform is learning from real IMS traffic instead of synthetic-only training data.

## Why This Lab Matters

This lab is where the demo becomes credible.

Instead of saying "the model was trained on synthetic sample rows," you can now say:

> The model was trained on feature windows derived from real SIP traffic generated by SIPp and processed by OpenIMSs.

That is the core message for customers and stakeholders. This lab establishes the traffic source, the scenario labels, and the dataset that the AI pipeline will use later.

## What Is Running In This Lab

- OpenIMSs core functions:
  - `ims-pcscf`
  - `ims-scscf`
  - `ims-icscf`
  - `ims-hss`
  - `openimss-webui`
- SIPp scenario runner built from this repo
- SIPp scenario files from `k8s/base/traffic/scenarios`
- `feature-gateway` for on-demand live feature-window generation
- MinIO object storage for captured dataset windows

## End-User Architecture Story

The simplest way to explain this lab is:

1. SIPp sends real scenario traffic into OpenIMSs.
2. The scenario runner captures what happened during that run.
3. The runner converts the run into a labeled feature-window JSON document.
4. The JSON document is written to MinIO under dataset version `live-sipp-v1`.
5. The same feature shape is later used by KFP training and by live anomaly scoring.

This keeps the live path and the training path aligned.

## Scenario Types

- `normal`
  - steady nominal traffic
  - expected label: normal
- `registration_storm`
  - high REGISTER pressure
  - expected label: anomaly
- `malformed_invite`
  - malformed INVITE traffic sent into the IMS entry point
  - expected label: anomaly

## Step-By-Step Walkthrough

1. Sync the `ims-demo-platform` Argo CD application or apply `k8s/overlays/demo`.
2. Confirm the IMS workloads are healthy:

```sh
oc get deploy -n ims-demo-lab
```

3. Verify the core IMS services are present:

```sh
oc get svc -n ims-demo-lab | rg 'ims-|openimss'
```

4. Verify the SIPp CronJobs exist and are enabled:

```sh
oc get cronjob -n ims-demo-lab | rg 'sipp'
```

5. Explain to the audience that these jobs are not only generating traffic. They are also writing labeled feature windows into MinIO for dataset version `live-sipp-v1`.
6. Trigger or wait for traffic generation, then confirm captured dataset windows exist:

```sh
oc create job --from=cronjob/sipp-normal-traffic sipp-normal-check -n ims-demo-lab
```

7. If you want to show an on-demand scoring path, call `feature-gateway/live-window/{scenario}` or use the demo UI. This demonstrates that live traffic can still be turned into a feature window immediately for scoring, while the scheduled jobs are building the training dataset in the background.

## What To Say During The Demo

- "This is real SIP traffic going into the IMS stack, not a mock endpoint."
- "Each scenario run becomes a labeled feature window in object storage."
- "Those stored windows are what the training pipeline uses later."
- "The online scoring path and the training path now share the same feature schema."

## What Success Looks Like

- IMS services are `Running`
- OpenIMSs WebUI is reachable through the `openimss-webui` route
- live traffic reaches the OpenIMSs P-CSCF on `ims-pcscf:5060`
- SIPp CronJobs persist labeled feature windows under the MinIO dataset prefix
- `feature-gateway` can still produce live feature windows for immediate scoring
- scenario definitions remain versioned in ConfigMaps instead of being embedded in ad hoc pods

## Key Message For The Audience

This lab proves that the platform is not separating "demo traffic" from "training data." The same real SIPp-driven IMS behavior is now feeding both the operational story and the machine learning story.
